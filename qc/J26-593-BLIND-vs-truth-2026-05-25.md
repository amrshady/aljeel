## Read of the Numbers

### 1. The 12-point drop (74.4% → 62.5%) is almost entirely the sponsorship gap.

When column N was pre-filled, the pipeline preserved Aljeel's "60307021.XXXXXX" combo for sponsorship rows via the `existing_account` preservation branch (process_batch.py L766-769). Strip column N, that pass-through vanishes — and the pipeline has no other signal for airline-ticket rows in sponsorship folders.

### 2. Travel-row performance is genuinely robust (90.6%).

The pipeline's name → Manpower → CC/DIV/Agency/Solution cascade works well on internal-employee travel. Blind ≈ prior here. The 10 misses are mostly segment-drift (correct account 60301003 but wrong cost-center because the traveler couldn't be uniquely resolved or the Manpower row carried unusual segments).

### 3. The 14 "real" employee-number cells are NOT recoverable from .msg evidence.

For those 14 rows, **Aljeel re-codes the trip to a DIFFERENT employee than the traveler.** Example: `ISMAIEL/YASIN MR` (1001406) flies, but Aljeel codes the GL row to `1000557` (Musaab — Yasin's peer under the same manager). The .msg body for this trip references ONLY Yasin (1001406) and Sanad (sfalshammari, gatekeeper). The number 1000557 appears NOWHERE in the evidence corpus. This is allocation policy living outside the document trail.

### 4. CC/DIV/Sol/Agency match Aljeel even when account is wrong.

For sponsorship rows like `ALKAF/FAHMI MR (6905084586)` the pipeline got CC=160014, DIV=170, Sol=10050, Agency=10072 **all correct** — only the account was 60301003 instead of 60307021. This means the segment plumbing IS finding the right team/agency via the family-cluster unification (Baassiri 1000820 booked the ticket → his Manpower row drives the segments). The single missing piece is flipping the account when the booking-employee is fronting a sponsorship event.

Plus an .msg `RE_ Barcelona Reservation May.msg`.
- **What we had:** Family-cluster unification correctly linked this airline ticket to voucher 26-622, which has emp_no 1000820 (Baassiri — internal Aljeel staff who booked the trip). Baassiri's Manpower row carried the right CC/DIV/Sol/Agency for an Abbott-HF event.
- **What we missed:** The presence of OPEX-HF-* PDFs in the SAME folder as the booking confirmation is a deterministic sponsorship signal. The pipeline's `_is_sponsorship(description)` only checks the line description string (e.g. "ALKAF/FAHMI MR - RUH IST BCN IST RUH (6905084586)") — which has no OPEX hint. **Folder-aware sponsorship detection is the missing layer.**
- **Fix:** if the ticket's raw folder (or any nearby folder linked by family-cluster) contains an OPEX-{HF,CRM,EP,SIS,DMS,DDW,IEPC}-* PDF, flag the row as sponsorship and set account=60307021 — but PRESERVE the already-derived CC/DIV/Sol/Agency.

## The Single Biggest Fix to Lift the Score

**Folder-aware sponsorship detection.** Net lift: ~34 of 37 sponsorship account flips = +21% on all-segments-exact (62.5% → ~83.5%).

Implementation sketch:
```python
def detect_sponsorship_from_folder(ticket_no, raw_dir):
    """Return True if the ticket's raw folder contains an OPEX-* PDF."""
    for d in find_msg_dirs_for_ticket(ticket_no, raw_dir):
        for p in d.glob("OPEX-*.pdf"):
            return True
        # Also check ancestor event folders by name pattern
        if re.search(r"(Reservation|Symposium|Meeting|Annual|Sponsor)", d.name, re.IGNORECASE):
            return True
    return False
```

Trigger inside `process_batch.py` right after the family-cluster unification: if `detect_sponsorship_from_folder` returns True, force account=60307021. Do NOT clobber CC/DIV/Sol/Agency — those are already correct via family-cluster.

Bigger but more invasive fix #2 (worth ~3-5 more rows): rewrite the OPEX-PDF Gemini prompt to constrain output to AlJeel-format 6-digit CC, 3-digit DIV, 5-digit Agency/Solution AND extract "requesting employee email/name" so we can resolve via Manpower.

Fixes that won't help much:
- Reading more of the .msg body — the bodies are short (5-15K chars), already fully read, and don't contain the recoded emp_no for the 14 internal-reallocation cases.
- Tweaking the email→emp_no cache — already at 677 entries and being used.
- More aggressive sponsorship keyword matching — the ticket-line descriptions genuinely don't have the keywords; the signal is at folder level.

The 11.9-point gap between blind and prior is approximately:

- ~2 pp from emp_no preservation on the 14 internal-reallocation rows

The pipeline's underlying intelligence — name resolution, family-cluster unification, CC/DIV/Sol/Agency derivation — is working as well in blind mode as it was in prior. The gap is entirely about places where the prior run was passing Aljeel's answer through.
